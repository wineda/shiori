栞（しおり）— デザイン＆仕様まとめ
パーソナルなジャーナリング（日記）アプリのデザインと仕様。プロトタイプ shiori-journal-v15.html に基づく。アプリ名は「栞（しおり）」、コンセプトは「明日の自分への伝言」。
1. コンセプト
「明日の自分への伝言」。今日という頁に栞を挟み、明日のわたしがひらく日記。「その場の呟き」と「一日の振り返り」の2層で、言葉だけを記録する。記録がたまると、履歴やうつろいタブのAI読み解きで、自分の流れを読み返せる。
呟き：思ったことを一言、即時に。低摩擦。
振り返り：一日をまとめて、明日の自分へ書き残す。保存は「今日に栞を挟む」。
キーワードは「静けさ」「余白」「明日へ託す」。
名前の由来：栞＝本の頁に挟んで、続きから読み始めるためのもの。今日を挟んで、明日ひらく。
2. デザイン原則
記録は言葉だけ：気分の入力（レーティング等）は置かない。書くことそのものに集中する。
唯一の色＝栞紅：UIは和紙色＋墨色の無彩色を基調とし、アクセントカラーは栞紐（スピン）の臙脂に由来する栞紅 #9C4A50 ただ一色。使いどころは「主ボタン・選択中状態・小さな印（呟きの点、ヒートマップ）」に限定し、本文・見出し・大きな面には使わない。
書体で2層を対比：呟き＝ゴシック（即時・軽い）、振り返り＝明朝（思索・ゆっくり）。文字そのものがモードを語る。
栞のメタファー：一日の記録を確定する行為を「今日に栞を挟む」と呼ぶ。
ロゴ：明朝体の一字「栞」＋その下に栞紅の短い傍線（案C）。ヘッダー左上に常置。アプリアイコンも同じ構成（和紙地に明朝「栞」＋紅の傍線）。
ミニマル：余白重視、装飾は最小限、角丸でやわらかく。
AIは人が最終確認：AIの出力は必ず"下書き/貼り付け"で、自動保存しない。記録に無いことは書かせない（グラウンディング）。

5. 機能仕様（画面別）
5.1 呟き
日付バー：画面上部の「‹ きょう ›」で表示日を切り替え。ラベルのタップで日付ピッカー。過去日を選ぶとラベルが栞紅になり、未来日は選べない。
直接入力：テキスト欄＋「のこす」。今日は時刻を自動付与。過去日を表示中はその日に追記でき、時刻の代わりに「✎」、「あとから」バッジが付く。
今日の呟きを新しい順に表示。
画像から取り込む：写真を選び、AIアプリ経由で文字起こし→貼り付け（§6）。
空のときは「まだ、しずかです」の空状態。
5.2 振り返り
今日集めた呟き：その日の呟きを一覧（内省の素材）。
入力方法は3つ：
直接入力（明朝の本文欄）
画像から取り込む（§6）
呟きからAI下書き（§7。呟きが1件以上ある時のみ有効）
本文を書いたら「今日に栞を挟む」で保存。
5.3 履歴
月カレンダー：記録件数（呟き数＋振り返り）に応じてセル背景を栞紅の濃淡4段階で濃くするヒートマップ。タップで詳細シート。今日は枠線で強調。凡例つき。
詳細シート：呟き一覧、振り返り、手書きバッジ、原本画像（タップで拡大）。
5.4 うつろい（読み解き）
機能はひとつだけ。シンプルに保つ。
期間切替：今週（7日）／今月（30日）／期間を選ぶ（開始日・終了日を指定）。
AIの読み解き：期間の記録をAIアプリに送り、読み解きを貼り付けて保存（§6）。今週/今月は前の同期間の記録も比較用に添える。任意期間は選んだ範囲のみ。明朝の文章で表示。「AIに送り直す」で更新。期間（任意期間は開始・終了の組）ごとにキャッシュ。
グラフやサマリーは置かない（複雑にしない方針）。
5.5 設定
リマインド（毎日の通知時刻。既定 21:00）のON/OFFと時刻。
AIプロンプトの編集：「今日の下書き」「今週の読み解き」「今月の読み解き」「任意の期間の読み解き」の4つを編集・保存できる。「既定に戻す」あり。
エクスポート：開始日・終了日を選び、その期間の呟きと振り返りを Markdown ファイルで書き出す（画像は含まない。ファイル名 shiori_開始_終了.md）。保存できない環境ではクリップボードへコピー。
サンプルデータの初期化（まっさらにする）。
6. AI連携（キー不要の「共有→貼り付け」ブリッジ）
方針
アプリにAPIキーもサーバーも持たせない。OSの共有シートで、ユーザーが持っている**AIアプリ（Claude / ChatGPT / Gemini など）**に文脈（画像やテキスト）とお願い（プロンプト）を渡し、返ってきた文章を貼り付けて取り込む。全AI機能がこの一つの操作体系に統一。
共通シート
「AIへのお願い（編集可）」＋「いっしょに送る内容」→［共有／まとめてコピー］→「結果を貼り付け」→反映。
共有時、お願いは常にクリップボードにも自動コピー（共有先が画像だけ受け取ってテキストを落としても、貼り付けられるように）。
機能別のプロンプト（初期値・編集可）
画像→呟き：「添付の手書きメモを、書かれている日本語のとおりに文字起こししてください。結果の本文だけを返してください。」
画像→振り返り：「添付の手書きの日記を文字起こしし、内容を数文でやさしく要約してください。結果の本文だけを返してください。」
呟きからAI下書き：今日の呟き（時刻・本文）を添えて、明日の自分へ宛てた一人称の振り返りを3〜5文で依頼（創作しない）。
うつろいの読み解き：期間の記録＋前期間の記録を添えて、流れ・変化・傾向をやさしくまとめる（記録に無いことは書かない）。
原則
AI出力は必ず下書き／貼り付け。自動保存しない。
記録に無いことは書かせない（プロンプトで明示）。
オンデマンド（明示操作でのみ）。
取り込み後も、しゅるい・日付を確認・修正できる。
7. データモデル & 保存
ストレージ
保存先：試作は window.storage（無い環境はメモリにフォールバック）。本番PWAでは IndexedDB に置き換える（§8参照）。抽象化された Store（get/set/del/list）の中身のみ差し替え、他コードは不変。
キー：
journal:day:YYYY-MM-DD … その日の記録
journal:settings … 設定（リマインド＋編集済みプロンプト4種：今日の下書き／今週／今月／任意期間）
journal:insight:{week|month}:{today} / journal:insight:custom:{from}:{to} … うつろいの読み解きキャッシュ
記録（day）の形
{
  "murmurs": [
    { "id": "…", "text": "…",
      "ts": 0, "time": "09:12",
      "img": "data:…|null", "source": "hand?", "late": true }
  ],
  "reflection": {
    "text": "…", "savedAt": 0,
    "img": "data:…|null", "source": "hand?"
  }
}
source: "hand" は画像取り込み由来。バッジ「✎ 手書き」を表示。
late: true は過去日にあとから追記した呟き。時刻は「✎」、バッジ「あとから」を表示。
取り込んだ画像は縮小（長辺640px, JPEG品質0.72）して添付保存。
うつろいのキャッシュ
{ "text": "読み解きの本文", "at": 0 }
8. PWA と保存・バックアップ（実装方針）
Claude Code での本番実装向けの方針。試作（HTML/CSS/JS 単一ファイル）を PWA 化する際の指針。
8.1 配布形態：PWA
HTML/CSS/JS 構成をそのまま PWA 化。ホーム画面インストール、オフライン起動、（インストール済みなら）通知に対応。ストア審査なしで配布可能。
必要ファイル：manifest.webmanifest（name=栞 / short_name=栞 / display: standalone / background_color theme_color は和紙 #F4F5F2 系 / アイコン各サイズ）、service-worker.js（アプリ本体をキャッシュしてオフライン起動）。
アイコン：ロゴC（明朝「栞」＋栞紅の傍線）。maskable を含む 192/512 等。
8.2 保存先：IndexedDB（最重要）
試作の window.storage を IndexedDB に置き換える。Store の get/set/del/list の実装だけ差し替えれば他コードは不変。
理由：
インストール済み PWA では、通常の「閲覧履歴・キャッシュ削除」で IndexedDB（サイトデータ）は消えない。Service Worker のキャッシュ（アプリ本体）とは別枠。
画像（base64）を含む大きめデータの保存に向く。日記アプリの標準的な保存先。
localStorage は使わない（容量が小さく画像に不向き）。
8.3 永続化の宣言
初回起動時に navigator.storage.persist() を呼び、ストレージを自動削除の対象外に要求。状態は navigator.storage.persisted() で確認。
navigator.storage.estimate() で使用量を把握（設定に表示してもよい）。
8.4 消える条件（正直な注意）
UI で軽く周知する。以下では消え得る：
ユーザーが明示的に「サイトデータも削除」した場合。
iOS Safari：インストールせずブラウザ利用かつ長期間（約7日）未使用（ITP）で消える可能性 → ホーム画面インストールの案内で回避。
端末の初期化・紛失・機種変 → 8.5 のバックアップで担保。
8.5 バックアップ／復元（最後の砦）
全データ JSON エクスポート：全 journal:* を1ファイルに書き出し（画像=base64 含む）。ファイル名 shiori_backup_YYYY-MM-DD.json。
JSON インポート：バックアップから復元（上書き／マージを選べると理想）。機種変時の移行手段。
役割分担：Markdown（§6）＝読む・共有する用、JSON＝丸ごと戻す用。
クラウド自動同期ではないため、定期バックアップ（ドライブ／iCloud へ保存）を促す一文を設定に置く。
8.6 通知（リマインド）
インストール済み PWA で Notification を利用。iOS は 16.4 以降＋ホーム画面追加が前提。
時刻リマインドは Service Worker と定期通知の制約に留意（プラットフォーム差あり）。まずは通知許可＋簡易スケジュールから。
8.7 将来：クラウド同期（スコープ外）
端末をまたいだ自動同期が必要になったら、ユーザー所有のクラウドや最小限のバックエンドを検討。現段階ではスコープ外。キー不要方針は維持。
9. その他の仕様
連続日数（ストリーク）：今日から遡って記録のある連続日数。今日が未記録でも途切れない。
リマインド：時刻設定を保存（実機ではローカル通知に対応させる想定）。
サンプルデータ：初回に過去約20日分を種まき（履歴・うつろいが最初から動く）。設定から初期化可。
画像ビューア：サムネイルのタップで原本を全画面表示。
トースト：操作結果を短く通知。
10. 決定の記録（なぜこうしたか）
AIは「共有→貼り付け」に統一：アプリにキーを埋め込むと漏れるため、直接API呼び出しはしない。サーバーも持たない方針に合わせ、OSの共有シート＋手持ちのAIアプリに委ねる形へ。プロトタイプからはAnthropic API直呼び出しを完全撤去。
PWA化と保存方針を決定：配布はPWA、保存はIndexedDB＋storage.persist()、機種変・紛失に備えたJSONバックアップ／復元を用意（§8）。ブラウザのキャッシュ削除では消えない設計に。
過去日への呟き追記を追加：呟き画面に日付バーを設け、入力し忘れた日にもあとから残せるように（「あとから」バッジで区別）。
うつろいに任意期間を追加：今週・今月に加え、開始日・終了日を選んで読み解けるように。
ブランド決定：アクセントカラーは栞紅 #9C4A50（栞紐の臙脂由来）、ロゴは明朝「栞」＋紅の傍線（案C）。候補比較は shiori-brand-board.html に記録。
お題（日替わりプロンプト）を廃止：振り返りの入口をさらに軽くし、下書きプロンプトからも {お題} の仕組みを撤去。
任意期間のエクスポートを追加：設定から期間指定で Markdown 書き出し。
履歴カレンダーを件数ヒートマップ化：黒丸ではなく、記録件数に応じて背景を濃くする濃淡表示に。
AIプロンプトを設定で編集可能に：今日／今週／今月の3プロンプトをユーザーが調整できるように。
空模様（気分＝天気）を廃止：感情データの入力自体をなくし、記録を言葉だけに。入力の手間を減らし、機能を複雑にしない方針。UIの色ドットは無彩色の印に置き換え。
うつろいタブを簡素化：気分の推移グラフとサマリー（記録日数など）を削除し、「今週/今月のAI読み解き」だけに絞った。機能を複雑にしない方針。
アプリ名を「そらもよう」から「栞（しおり）」に変更：コンセプトを「明日の自分への伝言」に再定義したため。
端末内AI／BYOKの検討結論：
端末内AI（iOS: Foundation Models / Android: Gemini Nano）は無料・オフライン・高プライバシーだが、対応端末が限られ、長文の合成（読み解き）は品質が落ちやすい。
まず「共有→貼り付け」で最小構成に。将来、OCRや軽い分類・要約を端末内に載せる、上級者向けにBYOKを足す、といった拡張余地を残す。
